# Core Beliefs

> 이 프로젝트의 기술적 신념. 원칙([CONSTITUTION.md](../../CONSTITUTION.md))보다 더 근본적인, 팀의 가치관을 기술한다.

---

## 1. 이동약자의 "불확실성"이 가장 큰 적이다

휠체어 진입 가능 여부, 경사로 유무, 장애인 화장실 위치 같은 정보는 "있는 것 같다/없는 것 같다"가 아니라 **검증된 상태 라벨**로 제공되어야 한다.
- 접근성 데이터는 `확인`/`미확인`/`부적합` 3단계로 정규화한다. "미확인"을 "가능"으로 암시하지 않는다.
- 출처(TourAPI/사용자 제보)를 UI에 반드시 노출한다.
- 알 수 없는 값을 긍정으로 대체하면 현장에서 직접 피해가 발생한다. 보수적으로 표현한다.

---

## 2. 외부 API 장애는 정상 상태다

TourAPI, 카카오맵, Gemini, 기상청 API는 모두 독립적으로 실패할 수 있다. 하나가 죽었다고 전체 서비스가 죽으면 안 된다.
- 각 외부 호출은 **개별 fallback**을 가진다: AI 설명 실패 → 기본 안내, 날씨 실패 → 생략, Kakao SDK 로드 실패 → 리스트 뷰로 degrade.
- 환경변수가 없어도 mock 데이터로 부팅 가능한 구조를 유지한다 (단, 프로덕션 필수 키는 `main.py` lifespan에서 검증).
- 캐시는 성능 최적화이자 장애 대비이다. Supabase 캐시 미스 시 파일 캐시, 파일 캐시 미스 시 인메모리로 폴백한다.

---

## 3. 로그는 코드보다 먼저 알려준다

사용자 피드백을 기다리기 전에 알고리즘/UX의 문제를 발견하려면 행동 로그가 있어야 한다.
- 추천 요청, 상호작용, 만족도 설문을 분리된 엔드포인트로 수집 (`/api/log/*`).
- Supabase로 적재하여 쿼리 가능한 상태로 보관한다.
- 로그 스키마 변경은 분석 파이프라인에 영향을 주므로 **한 번에 하나씩** 변경한다.

---

## 4. PWA 우선, 네이티브는 나중에

타겟 사용자(이동약자, 시니어 동반자)에게 "앱스토어 설치"는 높은 진입장벽이다. URL 공유 → 바로 사용이 최우선이다.
- 오프라인 fallback (`offline.html` + `sw.js`)을 유지한다.
- 공유 링크는 24시간 TTL을 가진 서버측 토큰 (`/api/share`)으로 발급한다.
- 모바일 브레이크포인트와 WCAG 2.2 AA 접근성을 기준선으로 한다.

---

## 5. 단일 책임의 라우터, 얇은 서비스

FastAPI 라우터는 HTTP 계약과 입력 검증만 담당한다. 비즈니스 로직은 `services/`로 분리한다.
- 라우터 하나는 하나의 리소스/동작을 다룬다 (recommend, share, weather, ...).
- 서비스 모듈은 외부 API 클라이언트 또는 순수 알고리즘이다. 라우터 간에 서비스는 자유롭게 재사용된다.
- 새로운 엔드포인트 추가 시, 기존 라우터에 끼워넣지 말고 새 파일을 만드는 쪽을 먼저 고려한다.
